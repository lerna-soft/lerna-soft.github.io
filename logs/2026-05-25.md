## Diagnóstico

Verifiqué el flujo end-to-end:
- Issue `#8064` de la sesión A se creó y procesó en 9s — el sync funciona.
- El `data.json` remoto del user tenía `progress["tt8594324"]`, `lastWatch`,
  y `history` con `playerStatus: 'playing'`. Correcto.
- El seed catalog `assets/catalog.seed.json` tiene la entry de "Ghosts" con
  `imdbId: tt8594324, tmdbId: null, posterUrl: <m.media-amazon URL>`.

**El user "navegador ghost" en realidad era la serie "Ghosts"** (TMDB 17174,
IMDb tt8594324). Confusión semántica afortunada porque me puso sobre la pista.

Hipótesis ordenadas por probabilidad:
1. GH Pages CDN cache lag — `loadRemoteWatchProgress` usaba
   `?v=${window.__mep_build}` que no cambia dentro de un mismo deploy.
2. Sesión B no autenticada con mismo email — `hydrateWatchProgressForCurrentUser`
   salía silencioso (sin log) cuando `getAuthUser()` retornaba null.
3. Cache de secciones del home pinta primero sin Continuar Viendo.
4. `scheduleAuthenticatedStartupWork` corre con ~3s delay + `requestIdleCallback`,
   no se redispara en visibilitychange.

**Bug secundario descubierto investigando** (el más insidioso):
- `getFilteredLocalTitles` filtra entries sin poster.
- `renderLocalCards` re-filtra por poster.
- `ensureCatalogEntriesFromWatchData` y `ensureCatalogEntryByHistory` crean
  entries SIN poster.
- Resultado: si en la sesión B el seed catalog no tiene la entry enriquecida
  (poster faltante), el título queda en `continueIds` pero los dos filtros lo
  excluyen del DOM. Invisible para siempre.

User dijo "hazlo todo" → procedí sin más preguntas.

## Implementación · 2 commits

### `2601b7c` feat(debug): logger por nivel + diag + cache bust + visibility rehydrate

- **`dtvLog`** ahora despacha por scope a `console.debug/info/warn/error` y
  usa `console.table` cuando data es tabular (array de objetos o objeto con
  primitivos). `boot/catalog/render → debug`, `sync/queue/player → info`.
- **`window.devilTvDiag()`** en DevTools: snapshot completo con
  `console.group/table`. Muestra auth, catálogo, synced progress, continueIds
  resueltos (con flags `inCatalog/hasPoster`), y warnings cuando hay ids
  huérfanos o sin poster. Pensado para diagnóstico mobile copy/paste.
- **`window.devilTvForceSync()`**: re-hidrata y vuelca diag.
- **`loadRemoteWatchProgress`**: `cacheTag = Date.now()` (no `__mep_build`).
  Antes, si el sync workflow escribía varias veces en un mismo deploy, el
  query string no cambiaba y el CDN servía stale.
- **`installVisibilityRehydrate()`**: re-hidrata cuando la pestaña vuelve a
  foreground >=60s después. Resuelve "abro mobile, está bloqueado, vuelvo y
  el Continuar Viendo no se actualizó".
- **`hydrateWatchProgressForCurrentUser`**: warn explícito cuando no hay
  auth user, en vez de retornar silencioso.

### `b50b17c` fix(continue-watching): triple fallback al catálogo completo

- **`resolveContinueWatchingItems(uniqueTitles, watch)`**: nuevo helper que
  resuelve `continueIds` con triple fallback:
  1. `uniqueTitles` (catalog filtrado).
  2. `loadLocalCatalog()` completo (incluye entries sin poster).
  3. Synced `progress[id]` (entry sintética con id como título).
- **`renderLocalCards(items, { allowMissingPoster })`**: opt-in al
  placeholder en HTML. El comportamiento default sigue siendo filtrar sin
  poster — solo "Continuar viendo" lo activa.
- **`buildSectionCarouselHtml`** y **`renderPaginatedLocalList`** propagan
  el flag solo para `def.key === 'continue'` / `sectionKey === 'continue'`.

## Tarde: análisis UX devil-tv + fix Servidor por defecto

User pidió análisis profundo de UX para devil-tv y reportó un bug específico:
"siempre que se abra el reproductor por defecto debe abrir la opción de
servidor 1 por defecto, pero a veces se abre la versión 4 y no se por que".

### Bug del Servidor 4 — causa raíz

`getActiveProvider` (`app.js:6472`) leía `mep_provider_v1` de localStorage.
`setActiveProvider` (`app.js:6817`) escribía cualquier cambio del user. La
primera vez que el user tocaba Servidor 2/3/4/5, ese valor quedaba pegado en
localStorage para siempre. Próxima apertura — otro título, otra sesión, otro
día — arrancaba con el provider pegado. Contradecía el comentario del propio
código en `app.js:234`: *"Servidor 1 (vaplayer) es el DEFAULT y no negociable
(sin ads)"*.

### Fix · commit `90a1c91`

`fix(player): reset al Servidor 1 en cada apertura fresca del reproductor`

- `getActiveProvider` y `setActiveProvider` ya no tocan localStorage.
- `openPlayerModal` resetea `state.activeProviderId` a Servidor 1 SOLO si
  `modal.hidden` al entrar. `jumpEpisode` reabre con el modal aún visible →
  conserva el provider que el user haya elegido durante la sesión actual.
- Residuo del key viejo en localStorage queda inerte (nadie lo lee).

Push: ya en `main`. GH Pages rebuild en curso.

### Análisis UX (delegado a Explore agent)

Reporte completo con 15 hallazgos priorizados (5 alto, 6 medio, 4 bajo).
Patrones transversales identificados:
- Errores de red sin UI diferenciada → user confunde fallo API con catálogo vacío
- Sync async invisible → "Continuar viendo" inconsistente entre dispositivos
- Mobile UX degradada en gestos/landscape (especialmente en player)
- Accesibilidad faltante → focus, aria, contraste

Top 5 (alto impacto):
1. **Búsqueda remota falla silenciosamente** (`app.js:4781`): user cree
   "no hay resultados" cuando es error de API. Falta toast diferenciado.
2. **Continuar viendo con títulos huérfanos sin poster** (`app.js:4961-4979`):
   `ensureCatalogEntryByHistory()` crea entries con `posterUrl: ''`. Lo
   parchamos parcialmente con `b50b17c` + `de5bf5c` pero quedan casos.
3. **Player fallback silencioso** (`app.js:6514-6528`): tras 6.5s sin
   evento, cambia iframe.src sin avisar. User no sabe por qué cambió.
4. **Títulos `playable: false`**: tratamiento inconsistente entre catálogo
   (pill-warn), detalle (botón disabled sin contexto) y búsqueda remota
   (filtro ignora playable).
5. **Sync de watch progress sin indicador**: `hydrateWatchProgressForCurrentUser`
   corre async ~3s sin spinner; datos viejos persisten silenciosos si falla.

Medio (6): regex laxo en reset.html, episodio activo poco visible en mobile,
help (?) de búsqueda sin handler claro, registro sin validador inline de
password, contraste bajo en hero con posters oscuros, player modal landscape.

Bajo (4): focus management en auth modals, timeout indefinido del catalog
loader (queda en 8%), feedback faltante en episodios sin metadata, datos
huérfanos en sync.

User dijo "hagamos algo con devil tv" — siguiente paso queda en sus manos:
priorizar cuáles atacar de los 5 de alto impacto. Tres de ellos son
relativamente baratos (toast en búsqueda, badge en fallback, copy en
playable:false). Los otros dos (Continuar viendo huérfanos, sync indicator)
son más estructurales.

### Fix adicional · botón fullscreen Servidor 4 — commit `d00cdf7`

`fix(player): allowfullscreen legacy en iframe para Servidor 4`

User reportó que el botón de fullscreen del Servidor 4 (111movies) no
funciona. El iframe ya declaraba `allow="autoplay; fullscreen; ..."`
(Permissions Policy moderna), pero algunos navegadores móviles y
específicamente 111movies (sin sandbox) exigen el atributo legacy
`allowfullscreen` para que el código interno del iframe pueda invocar la
Fullscreen API. Sin él, el botón pinta pero `requestFullscreen()` falla
silenciosamente.

Añadidos: `allowfullscreen` + `webkitallowfullscreen` + `mozallowfullscreen`
en `index.html:174-176`. Compatibles con todos los navegadores modernos
(el atributo se ignora donde no aplica).

Si después de probar sigue sin funcionar en Servidor 4, el siguiente paso
es chequear si el `requestNativeFullscreen(card)` que hacemos al abrir el
modal está bloqueando que el iframe pida su propio fullscreen (conflicto
de "qué elemento está actualmente en fullscreen").
