## Diagnóstico

Analicé el flujo end-to-end:
- `markCurrentSelectionAsStarted` (`app.js:6905`) guarda local
  `lastSeason`/`lastEpisode` y emite GitHub issue para sync remoto, pero
  SOLO si `!alreadyStarted`. Si el ep ya tenía `startedAt` local de antes,
  saltaba la actualización.
- `getResumeTarget` (`app.js:7951`) lee `progress.lastSeason/lastEpisode`
  para armar el label "Reanudar TxEy".
- Servidor 2/3/4/5 (vidsrc, 2embed, vidlink, embedmaster) **no emiten
  PLAYER_EVENT** — solo Servidor 1 (vaplayer) lo hace. Sin eventos, el
  único registro era el 'started' inicial, sin 'completed'.
- Drain workflow (`sync-watch-progress-from-issues.mjs`) procesa 1 issue
  por run (cada 5min). Si llegan 3 issues seguidos, los procesa
  secuencialmente. El último procesado **pisaba** lastSeason/lastEpisode
  sin chequear updatedAt order → un drain out-of-order podía retroceder.
- También: el workflow nunca actualizaba `watched{}` en el data.json
  remoto. Solo el frontend (localStorage) lo hacía. Resultado:
  getResumeTarget en otro dispositivo no podía avanzar al siguiente ep
  cuando uno se completaba.

## Implementación · 3 commits en devil-tv

### `68f0c9a6` fix(progress): siempre actualizar lastEpisode al abrir player

Movido el bloque que actualiza `lastSeason/lastEpisode` + emite
`queueWatchProgressSync` FUERA del `if (!alreadyStarted)`. El dedupe
interno por `(titulo:SxE:eventType)` con ventana 30min ya evita spam,
así que es seguro emitir siempre.

### `e39d4b98` feat(progress): heurística de tiempo para inferir completed

Nuevo state `state.playerSession = {openedAt, snapshot, sawEvent}` que se
inicializa en cada `openPlayerModal` y se flushea en `closePlayerModal` o
al cambiar de ep (jumpEpisode). Si `!sawEvent` y elapsed >= 22min
(series) / 80min (movie), emite issue `event_type: 'completed'` + marca
local. Umbrales conservadores para no marcar visto algo que solo quedó
cargando. Cuando el provider sí emite eventos reales, `sawEvent=true`
desactiva la heurística.

### `bfe5bc98` feat(sync): partición por título + respeto a updatedAt order + watched{} desde eventos

Tres cambios en `tools/sync-watch-progress-from-issues.mjs`:

1. **updatedAt order**: solo aceptar incoming si su `updatedAt >=
   previous.updatedAt` para pisar lastSeason/lastEpisode. Resuelve race
   condition del drain out-of-order.
2. **watched{} desde eventos**: nueva función `applyEventToWatched` que
   propaga `eventType: 'completed'` a `watched[sXeY].completedAt`. Ahora
   getResumeTarget en otro dispositivo puede avanzar al siguiente ep.
3. **Partición por título**: además del `data.json` monolítico
   (preservado como fuente principal), escribir:
   - `users/<email>/index.json` — overview ligero.
   - `users/<email>/titles/<imdbId>.json` — detalle por título.
   Habilita migración gradual del admin dashboard.
