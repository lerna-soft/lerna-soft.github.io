## Diagnóstico BudgetApp (2026-05-27, foco front, scan ligero back)

User pidió claridad sobre el estado actual de BudgetApp antes de
arrancar. Hice scan cross-repo: detallado del front, medio del back,
parent y mobile solo overview. Ambos repos del front y back tienen
último commit real del **21 de abril** (después solo "nested docs"
y links). Mobile último real del 25 de abril.

### Front (`budgetapp-frontend`)

- **Stack**: Next.js 15.3.3 App Router + React 19.1.0 + **JS puro sin
  TS** + CSS vanilla + SweetAlert2 11. HTTP nativo (fetch). Estado: solo
  hooks. Forms HTML nativos.
- **Páginas funcionales**: `/login`, `/register`, `/forgot-password`,
  `/reset-password`, `/` (dashboard), `/gastos` (CRUD), `/presupuesto`
  (wizard 5 pasos con varianzas plan-vs-real por periodo YYYY-MM),
  `/deudas`, `/bancos` (accounts + cards, multi-moneda COP/USD/EUR),
  `/herramientas/dividir-cuentas` (split-bill manager, 5100+ líneas),
  `/profile`.
- **API proxy**: `/api/*` con selector `BUDGETAPP_API_TARGET=backend|lambda`,
  `BACKEND_API_BASE_URL` default `http://localhost:3001`, soporte
  `LAMBDA_API_BASE_URL` + `BUDGETAPP_API_BASE_PATH`.
- **Endpoints consumidos** (vistos en código): auth (login/register/me/
  logout/forgot-password/reset-password), accounts, cards, budgets
  (+summaries/current/period), expenses, expense-categories, subcategories,
  debts, recurring-bills, savings-goals, split-bills, friends, tags,
  financial-reality (summary, bootstrap).
- **Auth**: middleware.js protege `/`, `/profile/*`, `/bancos/*`,
  `/deudas/*`, `/gastos/*`, `/herramientas/*`. Cookie `budgetapp_session`,
  redirect a `/login?next=<path>`. Valida sesión vía `/auth/me`.
- **Build/deploy**: solo Dockerfile dev (`npm run dev` en puerto 3000).
  Sin vercel.json, sin GH Actions, sin CI/CD.
- **Tests**: cero. Sin jest/vitest/playwright/cypress, sin __tests__.
- **Deuda visible**: `pg` y `nodemailer` en deps del front — probable
  residuo de cuando el front tenía lógica server-side de password
  reset. Hay `.env.example` con `DATABASE_URL` y `AUTH_SECRET` que
  sugiere lo mismo. Limpiar antes del deploy.
- **Veredicto**: ~85% del código UI listo. Bloqueadores reales para
  el MVP web son deploy + backend operativo, no features.

### Back (`budgetapp-backend`)

- **Stack**: Node 20 Lambda handlers + router manual (sin framework),
  JS puro, `pg` directo con raw SQL, JWT en cookies, nodemailer para
  password reset. Solo 2 deps prod (`pg` + `nodemailer`).
- **Entrada**: `src/handler.js` (Lambda) + `src/local-server.js` (dev).
- **Modelo de datos inferido** (de los repositories, sin SQL files):
  users, accounts, cards, debts, expenses, budgets, transactions,
  recurring_bills, savings_goals, split_bills, friends,
  expense_categories, expense_custom_subcategories, password_reset_tokens,
  countries.
- **Endpoints**: todos los que el front llama están implementados (auth
  completo + CRUD de los 10 dominios + financial-reality + health).
- **Deploy**: template.yaml SAM válido (Lambda 256MB / 15s + HTTP API
  + CORS localhost:3000). **Sin samconfig.toml**, **sin GH Actions**.
  Deploy es interactivo (`sam deploy --guided`).
- **BLOQUEADOR CRÍTICO**: no hay migraciones SQL. Sin
  `scripts/init-db.sql` o Prisma migrate, el backend no arranca contra
  DB vacía. El código asume tablas pre-creadas.
- **Otros faltantes**: jobs de cierre mensual documentados en
  `docs/jobs-y-alertas.md` pero no implementados. RBAC documentado pero
  endpoints solo validan presencia de user, no rol. Sin tests.
